OVERVIEW
--------
This document is the engineering implementation plan and progress tracker for
the NSOE system. It is organized into 4 phases over 24 months. Each task is
marked with a status indicator:

  [ ] = Not started
  [~] = In progress
  [x] = Completed
  [!] = Blocked / needs decision

Each phase builds on the previous. Do NOT proceed to coding tasks before
completing the design and review gates at the end of each phase section.

================================================================================
KNOWN RISKS AND OPEN DECISIONS
================================================================================

  RISK-001: Sovereign cloud availability
            India's NIC Cloud may not offer all required managed services
            (managed Kafka, managed Flink). May require self-managed installs.
            Decision needed: NIC Cloud vs AWS GovCloud India vs hybrid.
            Owner: Infrastructure lead | Due: Before Phase 1.4

  RISK-002: DPDP Act 2023 biometric provisions
            Collecting and processing face biometrics may require explicit
            data principal consent under DPDP unless exempt under national
            security provisions. Legal interpretation needed.
            Owner: Legal counsel | Due: Before Phase 1.5.6 (face recognition)

  RISK-003: Training data availability
            GNN risk scoring requires labeled data (known terrorist actors vs
            civilians). Such a dataset does not exist publicly. Must partner
            with MHA/IB for labeled data under strict access controls.
            Owner: ML lead + Program authority | Due: Before Phase 2.5.3

  RISK-004: Inter-agency data sharing legal framework
            No current legal framework mandates data sharing between IB, RAW,
            state police for this system. MoU negotiations required.
            Owner: Program authority | Due: Before Phase 4.2

  RISK-005: Edge node supply chain security
            Jetson Orin and camera hardware may be subject to supply chain
            risk (firmware tampering, counterfeit hardware). Requires hardware
            attestation program.
            Owner: Security lead | Due: Before Phase 3.1

  OPEN-001: Face recognition at public spaces — YES or NO?
            Current design limits face recognition to checkpoints. If program
            authority wants broader deployment, requires separate legal sign-off
            and ethical review.

  OPEN-002: Automated response actions — scope definition
            System currently raises alerts only. If automated response actions
            (e.g., locking a door, alerting patrol) are in scope for Phase 4,
            the architecture requires changes and additional legal authorization.
